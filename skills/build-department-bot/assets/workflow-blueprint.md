## 当前模式

- 模式：`{{MODE}}`
- 候选上限：{{CANDIDATE_COUNT}}

## 默认流程

```text
开始（接收用户消息、用户标识、可用的引用上下文）
  ↓
范围与意图分类
  ├─ greeting → 简短问候
  ├─ exact → 检索唯一 approved FAQ → 原样标准答案
  ├─ ambiguous / multi_intent → 检索真实 FAQ 标题 → 最多 {{CANDIDATE_COUNT}} 条候选
  ├─ quoted_selection → 按引用消息中的候选序号 → 原样标准答案
  └─ fallback / sensitive / context_uncertain → 固定兜底
  ↓
统一最终输出（仅一个面向用户的 response 字段）
  ↓
结束
```

## 节点契约

| 节点 | 输入 | 输出 | 失败处理 |
|---|---|---|---|
| 开始 | query、userid、引用上下文（若平台提供） | 原始输入 | 缺少 query → 固定兜底 |
| 分类 | 原始输入 | 内部标签 | 不确定 → `context_uncertain` |
| 知识检索 | query | approved FAQ | 0 条 → 固定兜底；多条 → 候选 |
| 候选生成 | FAQ 标准问题 | 编号列表 | 不足 {{CANDIDATE_COUNT}} 条则按实际数量，不补造 |
| 最终输出 | 标准答案/候选/兜底 | response | 不暴露内部字段 |

## 平台配置检查

- [ ] 知识检索仅使用获批知识集。
- [ ] 最终答案不经过二次自由改写。
- [ ] 候选标题来自检索结果，不由模型编造。
- [ ] 引用消息内容能否传给工作流已实测。
- [ ] 裸序号是否有会话状态已实测；不可靠时不猜。
- [ ] 群聊是否按用户隔离上下文已实测。
- [ ] 只有一个最终用户可见输出节点。
- [ ] 调试标签、关键词和检索评分不进入 response。

## 可选人工动作分支

只有在平台能力、权限和承接人均确认后添加：

`用户明确要求人工 → 用户确认（如需要） → 推送联系人/创建工单 → 成功提示`

动作失败时返回固定人工渠道；不得把“可能支持”当作已支持。
