# 知识库规范

## 最小字段

| 字段 | 必填 | 说明 |
|---|---:|---|
| `id` | 是 | 稳定唯一编号，如 `KB-001`；不向用户展示 |
| `question` | 是 | 一个独立、完整的标准问题 |
| `answer` | 是 | 经过批准、允许原样输出的答案 |
| `status` | 是 | `approved`、`draft`、`retired`；机器人只使用 `approved` |
| `owner` | 是 | 能确认答案准确性的责任人或部门 |
| `source` | 是 | 审批记录、FAQ 表或制度名称；无法提供则标记待确认 |
| `scope` | 建议 | 全员、地区、法人、角色或其他适用范围 |
| `updated_at` | 建议 | 最近批准或确认日期 |
| `aliases` | 可选 | 常见同义问法，仅用于检索，不改变答案 |
| `handoff` | 可选 | 该问题需要人工处理时的渠道 |

## 原子化规则

- 一个条目只解决一个用户意图。
- 问题应能脱离上下文单独理解。
- 答案不得依赖“以上”“如下”等缺失上下文。
- 含多个条件时，用清晰条款保留所有获批条件；不要拆掉影响结论的前提。
- 不合并不同来源的答案，除非内容负责人批准合并文本。

## 进入知识库前的门槛

只有同时满足以下条件的条目才能标记为 `approved`：

1. 答案由内容负责人确认；
2. 适用范围明确或明确标记为全员通用；
3. 无未解决冲突；
4. 不含不应向目标用户公开的个人或敏感信息；
5. 人工承接渠道可用；
6. 已加入至少一条测试用例。

## 文件选择

- 平台支持标准问答对时，优先使用标准问答对。
- 仅支持文档上传时，优先使用结构稳定的 DOCX 或 Markdown 转 DOCX，每条 FAQ 使用一致标题层级。
- 表格便于治理，但平台检索效果需实测；不要默认 Excel 最适合模型读取。
- 原始 PDF 适合作为来源，不一定适合作为最终 FAQ 库。扫描 PDF、跨页表格和复杂页眉容易降低检索质量。

## 冲突处理

发现冲突时，不允许模型自行选择。建立冲突记录，注明冲突条目、来源、影响范围、负责人和截止日期；在确认前让相关问题走固定兜底。
