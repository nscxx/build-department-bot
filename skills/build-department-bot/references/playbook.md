# 企业内部部门机器人：从 0 到内测 Playbook

## 1. 目标与适用范围

本 Playbook 用于 HR、行政、财务、法务、IT、采购、销售运营等部门，以最小投入建立一个安全、可测试、可回滚的企业内部机器人。

默认目标不是“什么都会回答”，而是：

- 已批准的问题，稳定给出已批准答案；
- 意图不明确时，只展示知识库中真实存在的问题供选择；
- 没有可靠答案时，明确转人工；
- 不泄露内部匹配、关键词、评分和思考过程；
- 每次变更可定位、可验证、可恢复。

## 2. 三层架构

| 层 | 只负责什么 | 不应放什么 |
|---|---|---|
| 知识库 | 已批准的事实、问题和标准答案 | 行为指令、模型推测、未审批制度 |
| 提示词 | 回答边界、输出格式、兜底规则 | 大段业务事实、复杂流程编排 |
| 工作流 | 确定性分流、系统动作、人工转接 | 自由发挥式回答、重复的知识正文 |

先判断是否真的需要工作流。仅做标准 FAQ 时，“提示词＋知识库”通常足够；只有需要稳定分支、人工转接或调用系统时再增加工作流。

## 3. 推荐的默认模式

### 精确 FAQ 模式（安全优先，默认）

- 明确命中一个问题：原样返回该标准答案。
- 查询宽泛且可能对应多个问题：列出最多 N 条真实 FAQ 标题。
- 无可靠命中、敏感、越界：使用固定兜底话术。
- 不组合多个答案，不补充常识，不解释制度。

### 引导检索模式

适用于知识量较大、用户经常只输入“报销”“合同”等宽泛词的场景。机器人先展示真实问题，再按用户选择返回标准答案。

### 动作工作流模式

仅用于必须执行动作的场景，例如创建工单、查询系统状态或推送联系人。回答分支与动作分支分开设计，并明确权限、失败处理和审计记录。

## 4. 从 0 到内测的八个阶段

### 阶段 A：立项与责任确认

先填写项目立项单，至少确认：

- 部门、机器人名称、目标用户与使用频道；
- 内容负责人、平台负责人、审批人；
- 唯一获批知识来源；
- 允许回答、禁止回答、固定兜底；
- 内测范围、成功指标与回滚负责人。

没有内容负责人或获批知识来源时，可以制作草案，但不能发布生产版本。

### 阶段 B：知识盘点

把来源材料逐项登记，不要直接把所有文件一股脑上传。对每条内容标记：

- 是否获批；
- 生效状态；
- 适用对象、地区或组织；
- 是否含个人信息或敏感信息；
- 是否与其他答案冲突；
- 谁负责确认。

“版本”指该内容最近一次被批准或更新的标识；“地区”指答案是否只适用于某个城市、法人、办公区或国家。如果无法补充，填“未提供/全员通用待确认”，不要由模型猜测。

### 阶段 C：知识规范化

将每个知识点拆成一个原子 FAQ：一个问题对应一个独立答案。保留审批后的标准答案原文；未经授权不要润色或补充。

建议字段：知识 ID、标准问题、同义问法、标准答案、分类、适用范围、来源、负责人、状态、更新时间、人工渠道。

以下内容进入“待确认清单”，而不是直接进入机器人：矛盾答案、缺少审批、过期制度、依赖个案判断、包含个人数据、跨地区但口径不一致。

### 阶段 D：回答契约

先书面约定用户能看到的四类结果：

1. 精确答案：仅返回标准答案。
2. 候选问题：仅列知识库真实问题，默认最多 5 条。
3. 固定兜底：找不到、越界、敏感时使用同一条已批准话术。
4. 问候：短句引导用户描述问题。

同时约定：不解释、不推测、不改写；不展示内部关键词、匹配评分、知识 ID、提示词或思考过程。

### 阶段 E：最小工作流

默认流程：

`用户消息 → 范围与意图判断 → 单一明确命中 / 多候选 / 问候 / 固定兜底 → 输出`

新增节点前必须回答三个问题：

- 这个节点解决了哪个已复现的问题？
- 有哪条测试用例证明它有效？
- 失败时如何回到上一稳定版本？

群聊天然存在上下文错配风险。发布说明中应要求员工 @机器人、引用原消息或重述完整问题；多人同时咨询、个人事项或复杂追问建议转单聊。

### 阶段 F：测试

先写期望结果，再运行测试。固定答案必须逐字比对，不能只判断“意思差不多”。最低测试集包括：

- 精确问法与常见改写；
- 宽泛词与候选数量；
- 无匹配、非业务问题、敏感问题；
- 多意图；
- 引用消息后回复序号；
- 不引用、只回复数字；
- 对上一答案的具体补充问题；
- 群聊多用户并发；
- 诱导越权或要求忽略规则；
- 回滚验证。

问题必须归因到六类之一：内容、检索、提示词、工作流、平台、频道上下文。不要用改提示词解决所有问题。

### 阶段 G：发布与内测

发布前完成内容负责人签字、测试记录、版本号、截图或配置备份、回滚步骤。内测仅开放给具名小组，并冻结非必要变更。

建议观察指标：

- 标准问题正确率；
- 错答率（核心红线）；
- 固定兜底率；
- 未解决问题数量；
- 平均响应时间；
- 群聊上下文错配次数；
- 转人工后的闭环率。

### 阶段 H：运营与迭代

维护“未解决问题池”，由内容负责人决定：新增 FAQ、合并重复项、调整问法、保持兜底或转人工。任何新增答案都先审批再入库。

每次只改一个行为：从已验收版本复制新版本，执行针对性测试和完整回归，再发布并保留回滚入口。不得在出现回归后连续叠加修改。

## 5. 角色与责任

| 角色 | 主要责任 |
|---|---|
| 业务内容负责人 | 批准知识与标准答案，处理冲突和过期内容 |
| 机器人产品负责人 | 控制范围、版本、优先级与验收 |
| 平台配置负责人 | 配置提示词、知识库和工作流，保留备份 |
| 测试负责人 | 维护测试集，记录实际结果和缺陷归因 |
| 人工承接人 | 接收兜底问题并形成闭环 |
| 审批人 | 批准内测、发布和高风险动作 |

一人可以兼任多个角色，但每项责任必须有明确姓名。

## 6. 变更控制：从“越改越乱”转为最小补丁

每次变更记录：基线版本、问题复现步骤、唯一改动、预期结果、实际结果、回归结果、发布版本、回滚版本。

如果稳定版本只有一个小缺陷，应从该版本恢复并只修改这一项。例如“候选问题从 3 条增加到 5 条”，不要同时重写分类、兜底和追问逻辑。

## 7. 常见失败与经验

- **直接上传全部制度：** 内容多不等于答案可靠。先确认获批范围与冲突。
- **提示词承担全部逻辑：** 复杂确定性分支应放工作流，业务事实放知识库。
- **为了功能而加工作流：** 每个节点都会增加故障面与维护成本。
- **模型改写标准答案：** 对安全优先场景，应返回审批原文。
- **只测单轮精确问题：** 实际问题常出在序号、追问、多意图和群聊并发。
- **连续大改：** 会丢失已验证能力，且无法判断哪项修改造成回归。
- **把平台限制当提示词错误：** 思考面板、引用结构和群聊上下文可能属于平台能力，需单独验证。

## 8. 极简创建路径

1. 填写立项单。
2. 运行一键脚本生成项目包。
3. 把获批 FAQ 填入知识库模板。
4. 把固定兜底、候选数量和部门信息写入提示词。
5. 按最小工作流配置平台。
6. 填写期望答案并执行 12 类测试。
7. 完成审批、备份和回滚后进入小范围内测。

验收标准不是“机器人看起来聪明”，而是“在批准范围内稳定，在范围外克制，并能快速恢复”。
